# 🍽️ Dish Scanner — Food101 Image Classifier

A from-scratch CNN trained on the [Food101](https://data.vision.ee.ethz.ch/cvl/datasets_extra/food-101/) dataset (101 food classes), served through a lightweight Python HTTP backend and a browser-based "dish scanner" UI.

No pretrained weights, no external inference frameworks — just a PyTorch CNN trained from zero, wrapped in a minimal stdlib `http.server` backend, and a vanilla JS/HTML/CSS frontend for uploading a photo and seeing predictions.

---

## How it works

```
┌─────────────────────────┐        ┌──────────────────────────┐
│  2_1-Food101_cnn_.py    │  saves │   Food101_CNN/            │
│  (training script)      │ ─────► │   food101_cnn.pth (+.bak) │
└─────────────────────────┘        └──────────────┬───────────┘
                                                    │ loads
                                                    ▼
┌─────────────────────────┐        ┌──────────────────────────┐
│  index.html / style.css │◄──────►│      server.py            │
│  app.js (browser UI)     │  HTTP  │  (inference server)       │
└─────────────────────────┘        └──────────────────────────┘
```

- **Training** (`2_1-Food101_cnn_.py`) trains the CNN on Food101 and checkpoints progress to disk so training can be safely paused and resumed.
- **Inference** (`server.py`) loads the latest checkpoint, exposes a tiny HTTP API, and serves the frontend files.
- **Frontend** (`index.html`, `style.css`, `app.js`) lets you drag/drop, browse, or paste an image, sends it to the server, and renders the predicted dish with a confidence bar and top-5 breakdown. All the model math happens in Python — the frontend only handles image I/O and rendering.

## Model architecture

**`HighCapacityFood101CNN`** — a 4-block convolutional network (~4.7M parameters):

| Stage           | Output shape         | Notes                                                   |
| --------------- | -------------------- | ------------------------------------------------------- |
| Input           | `(B, 3, 224, 224)`   | RGB image                                               |
| Block 1         | `(B, 128, 112, 112)` | Conv(64)→Conv(128), BatchNorm, MaxPool ÷2               |
| Block 2         | `(B, 256, 56, 56)`   | Conv(256)→Conv(256), BatchNorm, MaxPool ÷2              |
| Block 3         | `(B, 256, 28, 28)`   | Conv(256)→Conv(256), BatchNorm, MaxPool ÷2              |
| Block 4         | `(B, 512, 14, 14)`   | Conv(512), BatchNorm, MaxPool ÷2, Dropout(0.4)          |
| Global Avg Pool | `(B, 512, 1, 1)`     | Resolution-agnostic pooling                             |
| Classifier      | `(B, 101)`           | Linear(512→256) → ReLU → Dropout(0.3) → Linear(256→101) |

- **Optimizer:** AdamW (`lr=3e-4`, `weight_decay=1e-2`)
- **Scheduler:** `ReduceLROnPlateau` (patience=3, factor=0.5)
- **Augmentation (train only):** RandomCrop(224), RandomHorizontalFlip, ColorJitter
- **Expected performance:** ~48–58% top-1 accuracy at epoch 50 (typical ceiling for a scratch-trained CNN on Food101 — no transfer learning)

## Getting started

### 1. Install dependencies

```bash
pip install torch torchvision pillow
```

Use a CUDA-enabled build of PyTorch if you have a GPU — training on CPU is very slow for this dataset.

### 2. Train the model

Make sure `helper_function.py` (providing `accuracy_fn`) is in the same directory, then run:

```bash
python 2_1-Food101_cnn_.py
```

- Downloads Food101 automatically into `data/` on first run (~5GB).
- Trains in resumable chunks of 10 epochs at a time, prompting `Train for another 10 epoch(s)? (y/n)` after each chunk.
- Saves a checkpoint after every chunk to `Food101_CNN/food101_cnn.pth`, with an atomic write (temp file + rename) and automatic backup rotation (`food101_cnn.pth.bak`) so an interrupted save never corrupts your progress.
- Safe to stop at any point (`Ctrl+C` or answer `n`) — rerunning the script resumes from the last saved epoch.

### 3. Run the inference server

Once at least one checkpoint exists:

```bash
python server.py
```

This starts a server at **http://localhost:8000** which:

- Serves `index.html`, `app.js`, and `style.css`
- Loads the checkpoint (falling back to the `.bak` file if the primary is missing or corrupted)
- Exposes two endpoints:
  - `GET /api/status` — model readiness, device, epoch, parameter count, class count
  - `POST /api/predict` — accepts `{ "image": "<base64 data URL>" }`, returns the predicted dish, confidence, and top-5 candidates

### 4. Use the app

Open **http://localhost:8000** in a browser, then:

- Click the viewfinder to browse for a photo, drag-and-drop one in, or paste an image from your clipboard
- Click **"identify dish"**
- See the predicted dish, confidence bar, and top-5 ranked candidates, plus model metadata (epoch, parameter count, number of classes)

> Note: `app.js` currently points at `http://127.0.0.1:8000` explicitly rather than a relative path, so if you serve the frontend from a different host/port than the API, update the `API` constant at the top of `app.js` accordingly.

## Checkpoint format

Each checkpoint (`.pth`) is a dictionary containing:

```python
{
  "model_state_dict":     ...,
  "optimizer_state_dict": ...,
  "scheduler_state_dict": ...,
  "epoch": int,
  "loss": float,
}
```

`server.py` only needs `model_state_dict` plus `epoch`/`loss` for inference and metadata display, but the full state is preserved so training can resume exactly where it left off.

The 101 class names are hard-coded in `server.py` in alphabetical order — this matches the class-index ordering that `torchvision.datasets.Food101` assigns internally, so predictions map back to the correct label.

## Notes & gotchas

- The server's `HighCapacityFood101CNN` definition must stay in sync with the training script's — they're currently duplicated across both files rather than imported from a shared module.
- Preprocessing at inference time mirrors the training script's _test_ transform (`Resize(256)` → `CenterCrop(224)` → `Normalize` with ImageNet mean/std) — no random augmentation, so predictions are deterministic.
- If both the primary checkpoint and its backup are missing or corrupted, `server.py` will report `ready: false` via `/api/status` and the frontend will prompt you to train first.
