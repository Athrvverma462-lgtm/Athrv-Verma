# Text Transformer Classifier

A from-scratch PyTorch implementation of a Transformer encoder for text classification, trained on the [20 Newsgroups](http://qwone.com/~jason/20Newsgroups/) dataset.

Instead of using word embeddings or a pretrained tokenizer, this project takes a different approach: each document is first reduced to a fixed-size **TF-IDF vector**, and then every individual TF-IDF score is treated as a single scalar "token" fed into a standard Transformer encoder — similar in spirit to how tabular-transformer architectures treat each column of a table as a token. The idea was to explore what self-attention actually buys you over a plain feature vector, using an architecture I'd already built and understood for a different task.

## How it works

```
raw text
   │  TfidfVectorizer (fit on train set only)
   ▼
TF-IDF vector, shape (num_features,)
   │  reshape to (num_features, 1)
   ▼
Linear(1 → embed_dims)             # project each scalar feature into embedding space
   │
   + learned positional embedding  # so the model can tell features apart
   ▼
TransformerEncoder (self-attention + feedforward, N layers)
   │
   mean-pool over the sequence
   ▼
LayerNorm → Linear(embed_dims → num_classes)
   ▼
predicted newsgroup category
```

The model is trained with mini-batch gradient descent (`CrossEntropyLoss` + `Adam` + `ReduceLROnPlateau`), and checkpoints (model weights, optimizer state, epoch, loss) are saved to disk so training can be resumed later.

```bash
pip install torch scikit-learn
```

## Usage

Train from scratch for the default number of epochs:

```bash
python train.py
```

Train for a custom number of epochs:

```bash
python train.py --epochs 50
```

Resume training from the last saved checkpoint:

```bash
python train.py --resume --epochs 30
```

Keep training in repeated rounds, with a prompt after each one:

```bash
python train.py --interactive
```

Checkpoints are written to `checkpoints/text_transformer.pth`.

## Design notes / trade-offs

- **Why TF-IDF instead of a learned embedding layer or pretrained tokenizer?** Simplicity — it keeps the whole pipeline (vectorizer → tensors → model) self-contained and fast to iterate on, without downloading pretrained embeddings.
- **Why treat each TF-IDF feature as a "token"?** Mostly as an experiment in reusing a Transformer encoder — built originally for a different, genuinely sequential task — on tabular-style data, to see how self-attention over feature scores compares to just feeding the vector into an MLP.
- **Compute cost**: self-attention cost grows with `num_features²`, so `max_features` is deliberately kept small (100) to keep CPU training times reasonable. Raising it will noticeably slow down training.
- **This is a learning project**, not a benchmark-chasing one — a plain MLP on the same TF-IDF features would likely match or beat this in both accuracy and speed. The point here was understanding the architecture, not squeezing out state-of-the-art accuracy.

## Possible extensions

- Swap TF-IDF for learned word/subword embeddings and treat actual tokens (not TF-IDF scores) as the sequence.
- Compare against a plain MLP baseline on the same features.
- Add a `requirements.txt` / `pyproject.toml` for reproducible installs.
